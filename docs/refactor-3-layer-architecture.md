# Refactor: 3-Layer Architecture

## Context

The current `workersManager` instrumentation inside `packages/app` does two very different things: it queries the DB and coordinates jobs (orchestration logic), AND it directly spawns/manages C++ worker processes (process management). This conflation makes distributed deployment impossible.

The goal is a clean 3-layer split:

```
[packages/app]  workersOrchestrator  ←→  RabbitMQ  ←→  workersRunner [packages/runner]
                                                               ↕ stdin/stdout
                                                          workers [packages/workers]
```

Additionally, the dump file system (write JSON to disk, version with `latestDumpId`) is eliminated — the Start command now carries address data inline.

---

## Layer Naming

| Layer | Name in code | Package |
|-------|-------------|---------|
| 1 | `workersOrchestrator` | `packages/app` (instrumentation) |
| 2 | `workersRunner` | `packages/runner` (`@cybearl/celk-runner`, new) |
| 3 | workers | `packages/workers` (C++ binary, mostly unchanged) |

---

## RabbitMQ Queue Design

| Queue | Direction | Message types |
|-------|-----------|--------------|
| `celk.workers.commands` | orchestrator → runner | `Start`, `Stop` |
| `celk.workers.events` | runner → orchestrator | `Report`, `Match`, `Error` |

**Heartbeat + HeartbeatAck stay local-only** (runner ↔ worker via stdin/stdout, never touch RabbitMQ). The runner handles liveness; the orchestrator doesn't need to know about it.

If a worker exits unexpectedly, the runner publishes an `Error` event. On the next poll the orchestrator detects the address list is still enabled and re-sends a `Start` command.

### Package identity
```json
{
  "name": "@cybearl/celk-protocol",
  "description": "Shared protocol types and enums for the Celk worker system"

```

  `ADDRESS_PRIVATE_KEY_GENERATOR_SUPPORTS_RANGE`

- All worker message types: `StartWorkerMessage` (with `addressDumps: AddressDump[]`),
  `HeartbeatAckWorkerMessage`, `StopWorkerMessage`, `WorkerHeartbeatMessage`,
  `WorkerReportMessage`, `WorkerMatchMessage`, `WorkerErrorMessage`, `AddressClosestMatches`,
  `WORKER_MESSAGE_TYPE` enum, `WorkerUserOptions`

### File structure
```
packages/protocol/
├── package.json
├── tsconfig.json
└── src/
    ├── index.ts       re-exports everything
    ├── address.ts     ADDRESS_TYPE, ADDRESS_NETWORK, ADDRESS_PRIVATE_KEY_GENERATOR, AddressDump
```

### C++ sync script
The protocol package also contains a `sync` script that generates
`packages/workers/src/protocol.hpp` from the canonical TypeScript definitions, stamping it with:
```cpp
// THIS IS AN AUTOGENERATED FILE. DO NOT EDIT THIS FILE DIRECTLY.
// @cybearl/celk-protocol
```
This is the only file that uses copying — TypeScript consumers import the package directly.

## Protocol Change

`StartWorkerMessage` drops `addressesDumpFilePath` and gains `addressDumps: AddressDump[]` inline. Payload stays well under 1 KB for typical address lists.

`AddressDump` type is kept as-is. `generateAddressDumpObject` moves out of `dumps.ts` into the orchestrator (called in `_sync` to build the inline payload instead of writing to disk).

Both `packages/app` and `packages/runner` import all protocol types from `@cybearl/celk-protocol`. No duplication — the package is the single source of truth. The C++ header is regenerated via the sync script whenever the protocol changes.

### DB Schema changes — dynamicConfig

`heartbeatIntervalMs` and `heartbeatTimeoutMs` are currently hardcoded in `WORKERS_CONFIG`. Promote them into `dynamicConfig` so they are runtime-tunable:

**`db/schema/dynamicConfig.ts`**
- Add `workerHeartbeatIntervalMs: integer(...).notNull()`
- Add `workerHeartbeatTimeoutMs: integer(...).notNull()`

**New Drizzle migration**
- Add the two columns with sensible defaults (5000 / 15000)

The relevant worker fields (`workerReportIntervalMs`, `workerHeartbeatIntervalMs`, `workerHeartbeatTimeoutMs`) are passed in `StartWorkerMessage`. Orchestrator-internal fields (`workersManagerPollIntervalMs`, retry settings, limits, etc.) are never forwarded.

### dynamicConfig change detection

On each poll cycle the orchestrator compares the freshly-fetched `dynamicConfig.updatedAt` to the previously stored value. If it changed:
1. Publish `Stop` for every ID in `_activeListIds`
2. Clear `_activeListIds`
3. Continue with the normal sync — which immediately re-issues `Start` for all still-enabled lists with the updated config

This means any runtime change to `dynamicConfig` propagates to all runners within one poll interval, with no extra machinery.

## packages/runner — NEW package

### Package identity
```json
{
  "name": "@cybearl/celk-runner",
  "description": "Celk worker runner — bridges RabbitMQ commands to C++ worker processes"
}
```

### File structure
```
packages/runner/
├── package.json
├── tsconfig.json
└── src/
    ├── main.ts            entry point: connect AMQP, start consuming commands
    ├── config.ts          static local values only: binaryPath, matchesDir, stopGraceMs
    │                       (heartbeat params come from orchestrator via Start command, not here)
    ├── env.ts             runner's own PRIVATE_ENV: AMQP_URL, paths — mirrors app's env system
    │   (no protocol.ts — imports directly from @cybearl/celk-protocol)
    ├── lib/
    │   ├── controls.ts    spawnWorker, stopWorker, sendToWorker, listenToWorker
    │   │                   (ported from app's controls.ts, DB calls removed)
    │   ├── matches.ts     saveMatchLocally with plaintext key (safety net before AMQP publish)
    │   └── formats.ts     logger prefix helper
    └── amqp/
        ├── consumer.ts    consume celk.workers.commands → dispatch to controls
        └── publisher.ts   publish celk.workers.events (Report/Match/Error)
```

### Key behaviors
- `controls.ts` routes worker stdout: Heartbeat → local ACK; Report/Match/Error → publish to `celk.workers.events`
- Match flow: save locally (plaintext key) → publish to RabbitMQ → orchestrator encrypts + saves + DB + email
- Maintains `workers: Map<addressListId, ChildProcess>` for Stop routing and exit detection
- On unexpected worker exit: publish `Error` event so orchestrator can reschedule on next poll

### Workers package install scripts
- `install:dev` / `deploy` in `packages/workers/package.json`: update target path from app's binary location to runner's configured `binaryPath`

## Implementation phases

This refactor touches 4 packages (2 created from scratch), 2 DB migrations, and a C++ sync script. Implement one phase at a time -- each is independently buildable and reviewable.

**Phase 1 - @cybearl/celk-protocol**
Create the package, define all types and enums, write the sync script, run it to regenerate `packages/workers/src/protocol.hpp`, update `packages/app/db/schema/address.ts` to re-export enums from it. Update root `package.json` workspace shortcuts. Everything else depends on this being done first.

**Phase 2 - DB migrations**
Drop `latest_dump_id` from `address_lists`. Add `worker_heartbeat_interval_ms` and `worker_heartbeat_timeout_ms` to `dynamic_config`. Short and independently verifiable.

**Phase 3 - C++ cleanup**
Delete `src/core/dump.cpp/hpp` and `src/utils/files.cpp/hpp`. Update `src/main.cpp` to use `startMessage.addressDumps` inline. Update `CMakeLists.txt`. Build the binary and confirm it compiles cleanly before touching any TypeScript.

**Phase 4 - App refactor (workersOrchestrator)**
Rename the directory, delete `controls.ts` / `dumps.ts` / `protocol.ts` / `config/workers.ts`, rewrite `index.ts` with RabbitMQ publish/consume and `dynamicConfig` change detection, update env/queries. Largest single chunk.

**Phase 5 - @cybearl/celk-runner**
Create the new package, port `controls.ts` / `matches.ts` / `formats.ts` from the app (removing all DB calls), add AMQP consumer and publisher, update `packages/workers` install scripts to point here.

## Verification

1. Start RabbitMQ locally (Docker: `rabbitmq:3-management`)
2. Build workers binary: `yarn workspace @cybearl/celk-workers build`
3. Run `install:dev` from workers — confirm binary lands in runner's configured path
4. Start the runner: `yarn workspace @cybearl/celk-runner dev` — should connect to AMQP and wait
5. Start the app: `yarn workspace @cybearl/celk-app dev`
6. Enable an address list in the UI → orchestrator publishes `Start` → runner spawns worker → worker sends heartbeats/reports → runner publishes events → orchestrator updates DB stats visible in UI
7. Disable the address list → `Stop` command published → runner stops the worker
8. Confirm no dump files are written anywhere during the above flow
9. Confirm `latestDumpId` column is gone from DB after migration
